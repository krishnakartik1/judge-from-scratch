# Stage 9 — publishing pipeline

End-to-end workflow for cutting the four Stage 9 release artifacts
to Hugging Face. Designed to be safe (everything is dry-run by
default, real uploads behind explicit `--confirm` flags) and
re-runnable (each step is idempotent — re-running an upload
overwrites the same paths in the HF repo).

## Layout

```
publish/
├── README.md                           ← this file
├── model_cards/
│   ├── gemma4-social-bias-judge.md         (DPO primary)
│   ├── gemma4-social-bias-judge-sft.md     (SFT-only secondary)
│   ├── gemma4-social-bias-judge-gguf.md    (GGUF repo, both)
│   └── gemma4-social-bias-judge-pairs.md   (dataset)
├── modelfiles/                         ← generated by build_modelfile.py
│   ├── Modelfile.Q8_0
│   ├── Modelfile.Q5_K_M
│   ├── Modelfile.Q8_0-sft
│   └── Modelfile.Q5_K_M-sft
├── export_gguf.py                      ← Modal CPU; produces /vol/gguf/{dpo,sft}/
├── build_modelfile.py                  ← local; produces Modelfiles/
└── upload_hf.py                        ← Modal + local; uploads 4 HF repos
```

## Operator workflow

The order matters: GGUF repo goes up first so the Ollama one-liner
referenced in the model cards is live when the model cards land.

```bash
# 0. Pre-flight: confirm both merged checkpoints are on the volume.
uv run modal volume ls judge-from-scratch /checkpoints/ \
  | grep -E "merged-fp16|sft-merged-fp16"

# 1. Convert merged fp16 -> Q8_0 + Q5_K_M GGUFs (both models).
#    Runs on a Modal CPU container (~30 min, ~$0.10 total).
modal run publish/export_gguf.py

# 2. Generate Modelfiles. Asserts no <|think|> in the system prompt.
uv run python publish/build_modelfile.py

# 3. Dry-run every upload target. Inspect the planned file lists.
modal run publish/upload_hf.py::upload_remote --target gguf
modal run publish/upload_hf.py::upload_remote --target model-sft
modal run publish/upload_hf.py::upload_remote --target model-dpo
uv run python publish/upload_hf.py --target pairs

# 4. Upload for real (in the order the README declares).
modal run publish/upload_hf.py::upload_remote --target gguf      --confirm
uv run python publish/upload_hf.py        --target pairs          --confirm
modal run publish/upload_hf.py::upload_remote --target model-sft --confirm
modal run publish/upload_hf.py::upload_remote --target model-dpo --confirm
```

## Local llama.cpp install (only if you want to convert outside Modal)

The default path runs the conversion on Modal — no local llama.cpp
install required. If you do want to convert locally for development
or a one-off check, here's the minimum:

```bash
# Clone at the same tag the Modal image uses
git clone --depth 1 --branch b6800 \
  https://github.com/ggerganov/llama.cpp /opt/llama.cpp
cd /opt/llama.cpp
cmake -B build -DGGML_CUDA=OFF -DLLAMA_CURL=OFF -DBUILD_SHARED_LIBS=OFF
cmake --build build --target llama-quantize -j 4
uv pip install -r requirements.txt

# Then:
python /opt/llama.cpp/convert_hf_to_gguf.py \
  /vol/checkpoints/merged-fp16 --outfile out.gguf --outtype f16
/opt/llama.cpp/build/bin/llama-quantize out.gguf out.Q8_0.gguf Q8_0
```

The Modal path is preferred — the merged checkpoints are 16 GB each
and live on the project volume; pulling them locally just to
quantize is wasteful.

## Auth

`upload_hf.py` reads the HF token from (in order): `HF_TOKEN` env
var, `HUGGINGFACE_HUB_TOKEN` env var, `~/.huggingface/token` cache
(populated by `huggingface-cli login`). On Modal, the
`huggingface` Secret provides `HF_TOKEN` automatically.

## What gets uploaded where

| Target | HF repo | Repo type | Source |
|---|---|---|---|
| `gguf` | `krishnakartik/gemma4-social-bias-judge-gguf` | model | `/vol/gguf/{dpo,sft}/*.gguf` + local Modelfiles + card |
| `pairs` | `krishnakartik/gemma4-social-bias-judge-pairs` | **dataset** | local `data/{formatted,pairs}/*.jsonl` + card |
| `model-sft` | `krishnakartik/gemma4-social-bias-judge-sft` | model | `/vol/checkpoints/sft-merged-fp16/` + card |
| `model-dpo` | `krishnakartik/gemma4-social-bias-judge` | model | `/vol/checkpoints/merged-fp16/` + card |

Each upload synthesizes the repo's `README.md` from the
corresponding markdown file under `publish/model_cards/`. Edit
those four files to revise content; the upload scripts read them
fresh on each run.
